# Tomowa Access Patterns

## Purpose

This document defines the exact read and write patterns required by the Tomowa API.

The PostgreSQL schema, indexes, and application architecture should be designed around these access patterns rather than creating tables first and hoping they satisfy future queries.

---

# Core Entities

Tomowa currently has two core entities:

1. Session
2. Session Attendee

A Session represents one online language exchange event.

A Session Attendee represents one user joining one session.

---

# Access Pattern Summary

| API Endpoint                           | Access Pattern                        |
| -------------------------------------- | ------------------------------------- |
| `POST /api/sessions`                   | Create one session                    |
| `GET /api/sessions`                    | List upcoming sessions                |
| `GET /api/sessions/:sessionId`         | Get one session by ID                 |
| `PUT /api/sessions/:sessionId`         | Update one session if organizer       |
| `DELETE /api/sessions/:sessionId`      | Delete one session if organizer       |
| `POST /api/sessions/:sessionId/join`   | Join session if capacity allows       |
| `DELETE /api/sessions/:sessionId/join` | Leave session                         |
| `GET /api/me/sessions-created`         | List sessions created by current user |
| `GET /api/me/sessions-joined`          | List sessions joined by current user  |

# Detailed Access Patterns

## 1. Create Session

Endpoint

```text
POST /api/sessions
```

Inputs

```text
Authenticated user ID
Session form data
```

Writes

```text
One Session record
```

Business rules

- `organizerId` comes from the authenticated Supabase user.
- `sessionId` is generated by PostgreSQL.
- `createdAt` and `updatedAt` are generated by the backend.

---

## 2. Browse Upcoming Sessions

Endpoint

```text
GET /api/sessions
```

Inputs

```text
Current timestamp
Pagination limit
Pagination cursor
```

Reads

```text
Many Session records
```

Filtering

```text
startsAt >= current timestamp
```

Ordering

```text
Soonest sessions first
```

Notes

- Meeting links must not be returned.
- The database should have an index supporting this query.
- No filtering or recommendations in MVP.

---

## 3. View Session

Endpoint

```text
GET /api/sessions/:sessionId
```

Reads

```text
One Session
Possibly one SessionAttendee record
```

Authorization

```text
If user is organizer:
    include meetingLink

Else if user has joined:
    include meetingLink

Else:
    omit meetingLink
```

---

## 4. Update Session

Endpoint

```text
PUT /api/sessions/:sessionId
```

Reads

```text
One Session
```

Writes

```text
Update Session
```

Business rules

- Only the organizer may update.
- Refresh `updatedAt`.
- Capacity cannot become smaller than the number of attendees.

---

## 5. Delete Session

Endpoint

```text
DELETE /api/sessions/:sessionId
```

Reads

```text
One Session
```

Writes

```text
Delete Session
```

Business rules

- Only the organizer may delete.

Implementation

The `session_attendees` table should reference `sessions` with a foreign key.

For MVP, `ON DELETE CASCADE` is sufficient.

---

## 6. Join Session

Endpoint

```text
POST /api/sessions/:sessionId/join
```

Reads

```text
Session
SessionAttendee
```

Writes

```text
Insert SessionAttendee
```

Business rules

- User cannot join twice.
- Organizer cannot join their own session.
- Session must not exceed capacity.

Implementation

This operation should execute inside a PostgreSQL transaction.

---

## 7. Leave Session

Endpoint

```text
DELETE /api/sessions/:sessionId/join
```

Reads

```text
SessionAttendee
```

Writes

```text
Delete SessionAttendee
```

Business rules

- User must already be attending.
- Organizer cannot leave their own session.

Implementation

Execute inside a PostgreSQL transaction.

---

## 8. List Sessions Created By Me

Endpoint

```text
GET /api/me/sessions-created
```

Reads

```text
Many Session records
```

Filtering

```text
organizerId = authenticated user
```

Ordering

Upcoming sessions first.

---

## 9. List Sessions Joined By Me

Endpoint

```text
GET /api/me/sessions-joined
```

Reads

```text
SessionAttendee

↓

JOIN

↓

Session
```

Ordering

Upcoming sessions first.

# Authorization Requirements

## Is the user the organizer?

Needed by

```text
PUT /api/sessions/:sessionId
DELETE /api/sessions/:sessionId
GET /api/sessions/:sessionId
```

Uses

```text
Session.organizerId
```

---

## Has the user joined?

Needed by

```text
GET /api/sessions/:sessionId
DELETE /api/sessions/:sessionId/join
```

Uses

```text
SessionAttendee(sessionId, userId)
```

---

## Can the user view the meeting link?

Rule

```text
User may view meetingLink if:

- organizer
OR
- attendee
```

---

# Database Design Considerations

The schema should efficiently answer:

1. Get one session by ID.
2. List upcoming sessions.
3. List sessions created by one organizer.
4. List sessions joined by one user.
5. Check whether a user already joined.
6. Prevent overbooking.
7. Cascade attendee deletion when deleting a session.

# MVP Scope Guardrails

Not included in MVP:

- Search
- Language filtering
- Recommendations
- Direct messages
- User profiles
- Reviews
- Ratings
- Notifications
- Video calls
- Payments
- Recurring sessions

---

# Final MVP Requirements

```text
Create session

Browse sessions

View one session

Update own session

Delete own session

Join session

Leave session

List sessions I created

List sessions I joined

Protect meeting links

Prevent overbooking
```

Once these access patterns are implemented, the backend supports all required MVP functionality.
