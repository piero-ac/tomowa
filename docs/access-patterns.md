# Tomowa Access Patterns

## Purpose

This document defines the exact read and write patterns required by the Tomowa API.

The DynamoDB table design should be derived from these access patterns, not from a relational schema-first approach.

---

# Core Entities

Tomowa has two core pieces of data:

1. Session
2. RSVP / Attendance record

A session represents an online language exchange event.

An RSVP record represents a user joining a session.

---

# Access Pattern Summary

| API Endpoint                           | Access Pattern                        |
| -------------------------------------- | ------------------------------------- |
| `POST /api/sessions`                   | Create one session                    |
| `GET /api/sessions`                    | List upcoming sessions                |
| `GET /api/sessions/:sessionId`         | Get one session by ID                 |
| `PUT /api/sessions/:sessionId`         | Update one session by ID if organizer |
| `DELETE /api/sessions/:sessionId`      | Delete one session by ID if organizer |
| `POST /api/sessions/:sessionId/join`   | Create RSVP if capacity allows        |
| `DELETE /api/sessions/:sessionId/join` | Delete RSVP for current user          |
| `GET /api/me/sessions-created`         | List sessions created by current user |
| `GET /api/me/sessions-joined`          | List sessions joined by current user  |

# Detailed Access Patterns

## 1. Create Session

Endpoint:

```text
POST /api/sessions
```

Access pattern:

```text
Create one new session item.
```

Inputs:

```text
Authenticated user ID
Session form data
```

Writes:

```text
One Session record
```

Notes:

- `organizerId` comes from the Cognito user `sub`.
- `attendeeCount` starts at `0`.
- `createdAt` and `updatedAt` are generated by the backend.
- `sessionId` is generated by the backend.

---

## 2. Browse Upcoming Sessions

Endpoint:

```text
GET /api/sessions
```

Access pattern:

```text
List upcoming sessions ordered by start time.
```

Inputs:

```text
Current timestamp
Pagination limit
Pagination cursor
```

Reads:

```text
Many Session records
```

Filtering:

```text
startsAt >= current timestamp
```

Sort order:

```text
Soonest sessions first
```

Notes:

- Meeting links must not be returned.
- This access pattern likely requires a GSI for browsing upcoming sessions by start time.
- For MVP, no search, filters, recommendations, or language matching.

---

## 3. View Session

Endpoint:

```text
GET /api/sessions/:sessionId
```

Access pattern:

```text
Get one session by sessionId.
```

Inputs:

```text
sessionId
Authenticated user ID
```

Reads:

```text
One Session record
Possibly one RSVP record for this user and session
```

Authorization logic:

```text
If authenticated user is organizer:
  return session with meetingLink

Else if authenticated user has joined:
  return session with meetingLink

Else:
  return session without meetingLink
```

Notes:

- This requires efficient lookup of:
  - Session by ID
  - RSVP by session ID + user ID

---

## 4. Update Session

Endpoint:

```text
PUT /api/sessions/:sessionId
```

Access pattern:

```text
Update one session by sessionId if the authenticated user is the organizer.
```

Inputs:

```text
sessionId
Authenticated user ID
Updated session form data
```

Reads:

```text
One Session record
```

Writes:

```text
Updated Session record
```

Authorization logic:

```text
Only allow update if session.organizerId === authenticatedUserId
```

Notes:

- Meeting link can be updated.
- `updatedAt` should be refreshed.
- Capacity should not be updated below current `attendeeCount`.

---

## 5. Delete Session

Endpoint:

```text
DELETE /api/sessions/:sessionId
```

Access pattern:

```text
Delete one session by sessionId if the authenticated user is the organizer.
```

Inputs:

```text
sessionId
Authenticated user ID
```

Reads:

```text
One Session record
```

Writes:

```text
Delete Session record
Possibly delete associated RSVP records
```

Authorization logic:

```text
Only allow delete if session.organizerId === authenticatedUserId
```

MVP decision:

```text
Delete the Session record.
RSVP cleanup may be handled simply for MVP.
```

Notes:

- DynamoDB does not automatically cascade deletes.
- For MVP, we can either:
  - Delete RSVP records when deleting a session, or
  - Treat deleted sessions as unavailable in joined-session views.
- Prefer the simplest implementation that keeps the user experience correct.

---

## 6. Join Session

Endpoint:

```text
POST /api/sessions/:sessionId/join
```

Access pattern:

```text
Create an RSVP record for the authenticated user and increment attendeeCount.
```

Inputs:

```text
sessionId
Authenticated user ID
```

Reads:

```text
One Session record
Possibly one RSVP record to check if user already joined
```

Writes:

```text
One RSVP record
Update Session attendeeCount
```

Business rules:

```text
User cannot join the same session twice.
Organizer cannot join their own session.
Session cannot exceed capacity.
User cannot join a deleted session.
```

Important DynamoDB requirement:

```text
This should use conditional writes or a transaction to avoid overbooking.
```

Notes:

- The clean implementation is likely a DynamoDB transaction:
  - Condition check session is not full
  - Create RSVP only if it does not already exist
  - Increment attendeeCount
- This is one of the most important production-quality backend behaviors in the MVP.

---

## 7. Leave Session

Endpoint:

```text
DELETE /api/sessions/:sessionId/join
```

Access pattern:

```text
Delete the authenticated user's RSVP record and decrement attendeeCount.
```

Inputs:

```text
sessionId
Authenticated user ID
```

Reads:

```text
One RSVP record
```

Writes:

```text
Delete RSVP record
Update Session attendeeCount
```

Business rules:

```text
User can only leave if they have joined.
Organizer cannot leave because organizer is not an attendee.
attendeeCount should not go below 0.
```

Important DynamoDB requirement:

```text
This should use conditional writes or a transaction.
```

Notes:

- Use a transaction to:
  - Delete RSVP only if it exists
  - Decrement attendeeCount safely

---

## 8. List Sessions Created By Me

Endpoint:

```text
GET /api/me/sessions-created
```

Access pattern:

```text
List sessions where organizerId equals the authenticated user ID.
```

Inputs:

```text
Authenticated user ID
Pagination limit
Pagination cursor
```

Reads:

```text
Many Session records
```

Sort order:

```text
Most relevant order for MVP:
Upcoming sessions first or newest created first
```

Notes:

- This likely requires a GSI on organizerId.
- For MVP, choose the simplest useful sort order.

---

## 9. List Sessions Joined By Me

Endpoint:

```text
GET /api/me/sessions-joined
```

Access pattern:

```text
List sessions the authenticated user has joined.
```

Inputs:

```text
Authenticated user ID
Pagination limit
Pagination cursor
```

Reads:

```text
Many RSVP records for the user
Then corresponding Session records
```

Sort order:

```text
Upcoming joined sessions first if possible.
```

Notes:

- This access pattern requires efficient lookup by attendee user ID.
- A GSI on RSVP records by user ID is likely needed.
- Depending on the DynamoDB design, joined session summaries may be denormalized into RSVP records to avoid many follow-up reads.
- For MVP, correctness and simplicity matter more than perfect optimization.

# Authorization Access Needs

The backend must efficiently answer these questions:

## Is the user the organizer?

Required by:

```text
PUT /api/sessions/:sessionId
DELETE /api/sessions/:sessionId
GET /api/sessions/:sessionId
```

Needed data:

```text
Session.organizerId
```

---

## Has the user joined the session?

Required by:

```text
GET /api/sessions/:sessionId
DELETE /api/sessions/:sessionId/join
```

Needed data:

```text
RSVP record for sessionId + userId
```

---

## Can the user see the meeting link?

Required by:

```text
GET /api/sessions/:sessionId
```

Rule:

```text
User can see meetingLink if:
  user is organizer
  OR user has joined the session
```

# DynamoDB Design Questions

The DynamoDB model should answer these questions:

1. How do we get one session by ID?
2. How do we list upcoming sessions by start time?
3. How do we list sessions created by one user?
4. How do we list sessions joined by one user?
5. How do we check whether a user already joined a session?
6. How do we prevent overbooking when multiple users join at the same time?
7. How do we keep `attendeeCount` correct?
8. How do we handle deleted sessions?

# MVP Scope Guardrails

The following access patterns are intentionally not supported in MVP:

- Search sessions by text
- Filter by language
- Match users by language
- Recommend sessions
- Chat between users
- Direct messages
- Reviews
- Ratings
- User profiles
- File uploads
- Avatars
- Video calls
- Payment flows
- Recurring Sessions

If a future feature requires one of these, it should be treated as Phase 2 or later.

---

# Final MVP Data Requirements

The MVP needs to support:

```text
Create session
Browse upcoming sessions
View one session
Update own session
Delete own session
Join session
Leave session
List sessions I created
List sessions I joined
Protect meeting links
Prevent overbooking
```

Once these access patterns are satisfied, the DynamoDB design is good enough for the MVP.
